## 概要

本プロジェクトのBisector Method実装において、工具中心座標(P)からプログラム座標(O)への変換時のZ方向オフセット量を、加工する形状タイプによって条件分岐させる実装が有効であることを確認しました。

**実装ルール**:
```typescript
const dz = isConvex ? 0 : noseR  // 凸円弧: 0、凹円弧・直線: noseR
```

**重要**: この条件分岐は、本プロジェクトの特定のBisector実装が返す「P座標の性質」に依存しています。他のアルゴリズムやCAMソフトには適用できない可能性があります。

## 問題の発見

### 初期の症状

角R0.5mm、ノーズR0.4mmの凸円弧において、従来の一律オフセット(`dz = noseR`)を使用した場合：

- **期待値（手書き計算）**: Z = -114.827mm
- **実測値（従来実装）**: Z = -115.193mm
- **系統的誤差**: **-0.366mm** ≈ -noseR

この誤差は、ノーズR値とほぼ同じ大きさで、Z方向に過補正が発生していた。

### 修正後の結果

条件付きdz実装(`dz = isConvex ? 0 : noseR`)に変更後：

- **実測値（修正後）**: Z = -114.793mm
- **誤差**: **+0.034mm** (91%改善)
- **全テスト**: **97個全てパス**

## 理論的背景

### 1. 座標系と変換の定義

#### 仮想刃先点（Imaginary Tool Tip）

工具ノーズ半径に対してX軸・Z軸平行な接線を引き、その交点を**仮想刃先点 O**と定義する。これはプログラミングの基準点となる仮想的な点であり、実際の切削点ではない。

#### 工具中心点P の定義（本実装特有）

**重要**: 本ドキュメントにおける「工具中心P」は、純粋な幾何学的工具中心ではなく、**本プロジェクトのBisector Method実装（`CenterTrackCalculator`クラス）が計算するノード座標**を指します。

この座標は：
1. 各セグメントを法線方向にノーズR分オフセット
2. 隣接するオフセット線の交点（bisector計算）で得られる
3. 形状タイプによって、既に一部のオフセット成分を含む可能性がある

**一般的な「工具中心」とは異なる可能性があることに注意してください。**

#### V_offset（仮想刃先点オフセット）

本実装におけるPから仮想刃先点Oへの変換式：

```
O = P - V_offset
```

Tool Tip 3（外径/前向き）の場合：
```
V_offset.x = noseR × 2  (直径値換算)
V_offset.z = dz  (条件付き - 本実装特有)
```

**注意**: 標準的なCNC文献における V_offset は形状によらない定数ですが、本実装では形状タイプに応じて dz を変化させています。これは本実装のP座標の性質に起因します。

### 2. なぜ条件付きdzが必要か（本実装における観察結果）

#### 標準CNC文献との視点の違い

**標準CNC文献（G41/G42補正）**:
- プログラム座標（仮想刃先点）を入力
- CNC制御が内部で工具中心を計算
- V_offsetは**O → P**方向の変換
- V_offsetは形状によらない定数

**本Bisector Method実装**:
- 本実装特有のP座標を計算
- プログラム座標に変換して出力
- V_offsetは**P → O**方向の変換
- **本実装では、Pの性質により条件分岐が有効**

#### ⚠️ 一般化の危険性

**本実装での観察**: 凸円弧で `dz = noseR` を使用すると約 -0.366mm の誤差が発生

**ただし、これは以下の理由による可能性があります**:
1. 本実装のBisectorアルゴリズムが、凸円弧のノードで特殊な計算を行っている
2. Pの定義自体に、既にZ方向成分が含まれている
3. 法線ベクトルの計算方法が特殊

**他の実装では異なる結果になる可能性が高い**ため、この条件分岐を一般的な「Bisector Method の性質」として扱うことは危険です。

#### 本実装での観察結果

##### 凸円弧（角R）の場合

**本実装での観察**:
1. 一律 `dz = noseR` を適用すると、約 -noseR の系統的誤差が発生
2. `dz = 0` にすることで、誤差が ±0.034mm に改善
3. ワーク円弧: 中心(Cx, Cz)、半径 Rw
4. 本実装が計算するP座標は、半径 Rw + noseR の円弧上に配置される

**推測される理由**:
- 本実装のBisector計算が、凸円弧のノードで特殊な処理を行っている可能性
- P座標が既に一部のZ方向オフセットを含んでいる可能性

**注意**: この挙動は本実装特有であり、他のBisector実装では異なる可能性があります。

##### 直線の場合

**本実装での観察**:
1. `dz = 0` では理論値とズレが大きい（例：45度で0.517mm）
2. `dz = noseR` で誤差が改善（例：45度で0.117mm）

**注意**: 「完全な V_offset = (noseR, noseR)」という表現は不正確です。実際は法線方向のオフセットであり、角度により成分比が変わります。

##### 凹円弧（隅R）の場合

**本実装での観察**:
1. `dz = noseR` が有効
2. 工具中心パス: 半径 Rw - noseR

**注意**: 凸円弧との違いの理由は、本実装のアルゴリズム詳細に依存します。

## 実装ガイドライン

### pToO関数の実装

```typescript
/**
 * 工具中心座標(P)からプログラム座標(O)への変換
 *
 * @param px - 工具中心X座標（直径値）
 * @param pz - 工具中心Z座標
 * @param noseR - ノーズ半径
 * @param toolType - 仮想刃先点番号（1-4, 8）
 * @param isConvex - 凸円弧かどうか（true: 角R, false: 隅R・直線）
 * @returns プログラム座標 {ox, oz}
 */
export function pToO(
    px: number,
    pz: number,
    noseR: number,
    toolType: number,
    isConvex: boolean = true
): { ox: number; oz: number } {
    let dx = 0, dz = 0

    switch (toolType) {
        // 凸コーナー（角R）: bisector法が既にZ方向を処理済み
        // 凹コーナー（隅R）・直線: 完全なZ方向オフセットが必要
        case 3: dx = noseR; dz = isConvex ? 0 : noseR; break;    // 外径/前向き
        case 4: dx = noseR; dz = isConvex ? 0 : -noseR; break;   // 外径/奥向き
        case 2: dx = -noseR; dz = isConvex ? 0 : noseR; break;   // 内径/前向き
        case 1: dx = -noseR; dz = isConvex ? 0 : -noseR; break;  // 内径/奥向き
        case 8: dx = noseR; dz = 0; break;                        // 端面
        default: dx = noseR; dz = isConvex ? 0 : noseR;
    }

    const ox = px - (dx * 2)  // 直径値換算
    const oz = pz - dz

    return { ox: round3(ox), oz: round3(oz) }
}
```

### isConvex判定ロジック

```typescript
// セグメントタイプに基づく判定
const isConvex = (seg.type === 'arc' && seg.isConvex !== false)

// true  → 凸円弧（角R）のみ → dz = 0
// false → 凹円弧（隅R）、直線 → dz = noseR
```

### 呼び出し例

```typescript
const calc = new CenterTrackCalculator(noseR, isExternal, toolType)
const compensated = calc.calculate(profileSegments)

// 内部でpToO関数が各ノードで呼び出される
// セグメントのisConvexフラグに基づき、自動的に適切なdzが適用される
```

- 法線ベクトルの計算方法に依存

このため、**一般化して文献に記述することが困難**です。

### 2. 標準的なアプローチとの違い

**標準的なCNC文献**:
- V_offsetは形状によらない定数として扱う
- O→P変換（ユーザー視点）で記述
- 本実装のような「形状タイプによる条件分岐」は想定外

**本実装**:
- P→O変換（実装視点）
- Pの性質が特殊なため、条件分岐が必要になった可能性

### 3. 商用CAMソフトの非公開性

- 商用CAMソフトは内部アルゴリズムを公開しない
- 他のCAMソフトが同じアプローチを取っているかは不明
- 実装詳細は企業秘密

### 4. 一般解の存在

**ChatGPT検証での指摘**:
- 一般解は法線ベースで記述すべき（形状タイプによらない）
- 本実装の条件分岐は「特定実装の最適化」に過ぎない可能性
- 文献は一般解を記述するため、本実装のような特殊ケースは含まれない

**結論**: 本実装ノートは「特定実装での観察結果」であり、一般理論としての文献記述には不適切である可能性が高い。
