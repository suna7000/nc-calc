# 座標系と変数の定義 ⭐

## 旋盤座標系

```
        Z軸（主軸方向）
         ←
    ─────┼─────
         │     X軸
         ↓   （半径方向）

X: 半径で管理（直径はX×2）
Z: 主軸方向の絶対位置（右が0、左がマイナス）
```

## 本実装における重要な定義

### 1. θ（シータ）の定義

**本実装では2種類のθを使用**：

#### θ_segment: セグメント角度
```typescript
// テーパー線の水平からの角度
seg.angle  // 例: 30° (度単位)
```

#### θ_normal: 法線間の角度
```typescript
// 2つの法線ベクトル間の角度
const dot = n1.nx * n2.nx + n1.nz * n2.nz
// dot = cos(θ_normal)
```

**重要**: どちらのθを使っているかを常に明確にする

### 2. dist の定義

**本実装におけるdistの意味**：

```typescript
// ref点（セグメント端点）からノードP点までの距離
// bisector方向での距離

P点 = ref点 + bisector単位ベクトル × dist

具体的には:
px = refX + bisec.bx * (dist / R) * R
pz = refZ + bisec.bz * (dist / R) * R
```

**注意**: これは「頂点から工具中心までの距離」とは異なる定義

### 3. fz の定義

**ユーザー記事におけるfz**：

```
fz = 幾何プロファイル座標から工具中心座標への
     Z方向の合計オフセット量

含まれるもの:
- 法線方向の垂直オフセット成分
- テーパー角度に応じた補正成分
- その他すべての補正成分の合計
```

**重要**: fzは「追加の補正」ではなく「合計補正量」

### 4. dz の定義

**本実装におけるdz**：

```typescript
// P座標からO座標を計算する際のZ方向オフセット
oz = pz - dz

P座標: 中間計算用の座標
O座標: 最終的な工具中心座標
```

## 原因2: Bisector距離計算式 ⚠️ 定義依存

### 問題の整理

**修正前の式**：
```typescript
const dist = this.noseR / Math.max(0.01, cosHalf)
```

**修正後の式**：
```typescript
const tanHalf = sinHalf / Math.max(0.01, cosHalf)
const dist = this.noseR * tanHalf
```

### ⚠️ 重要な注意

**これらの式の正誤は、以下に依存します**：

1. **θの定義**：
   - 法線間の角度（dot = cos θ）
   - cosHalf = sqrt((1 + dot) / 2) なので θ/2 は法線間角度の半分

2. **distの定義**：
   - ref点からP点までの bisector方向距離
   - P = ref + bisector × dist

3. **目的**：
   - ユーザーの手書き計算との一致

### 数値的検証

現行実装の定義では：

| θ_normal | R/cos(θ/2) | R×tan(θ/2) | 実測で一致 |
|----------|------------|------------|-----------|
| 90° | 1.414R | 1.000R | tan側 ✅ |
| 60° | 1.155R | 0.577R | tan側 ✅ |

**結論**：
```
「R×tan(θ/2)がより正確」ではなく
「現行実装の定義と目的では、R×tan(θ/2)が
 ユーザーの手書き計算と一致した」

他の定義では R/cos(θ/2) や R/sin(θ/2) が
正しい場合もある
```

### なぜ tan(θ/2) で一致したか

**仮説**：

ユーザーの記事の `fz = R × (1 ± tan(θ_segment/2))` と、
bisector計算の `dist = R × tan(θ_normal/2)` が、
特定の条件下で一致している可能性。

**今後の課題**：
この一致が偶然か、理論的根拠があるかの検証が必要。

## 原因3: テーパー専用補正式の不在 ✅ 機能不足

### 問題
テーパー線（0° < angle < 90°）に対する特別な処理がなかった

### ユーザーの記事の数式

```
上りテーパー（直径減少）: fz = R × (1 - tan(θ_segment/2))
下りテーパー（直径増加）: fz = R × (1 + tan(θ_segment/2))
```

**重要な発見**: 判定は**Z座標の変化ではなく、直径（X座標）の変化**

```typescript
// ❌ 誤り
const isDescending = (seg.endZ < seg.startZ)

// ✅ 正しい
const isDiameterIncreasing = (seg.endX > seg.startX)
```

### 修正: 新関数の実装

```typescript
function calculateDzForTaper(
    angle: number,              // θ_segment（度）
    noseR: number,
    tipNumber: number,
    isDiameterIncreasing: boolean
): number {
    if (tipNumber === 8) return 0

    const dzSign = [0, -1, +1, +1, -1]
    const sign = dzSign[tipNumber] || +1

    const thetaRad = (angle * Math.PI) / 180
    const halfAngleRad = thetaRad / 2

    // ユーザー記事の数式を実装
    const factor = isDiameterIncreasing
        ? (1 + Math.tan(halfAngleRad))
        : (1 - Math.tan(halfAngleRad))

    return noseR * factor * sign
}
```

**評価**: これはユーザーの実績ある計算式の実装。妥当。

## 原因4: Bisectorの直線への適用 ⚠️ 設計判断

### 問題の発見

```
入力: ΔZ=-0.347, ΔX=-0.200 (30°テーパー)
出力: ΔZ=-0.003, ΔX=-0.686 (ほぼ水平！) ❌

テーパー線が歪んでいた
```

### 根本原因

Bisector Methodは各ノードを独立に計算：
- 始点: 前セグメントとのBisector
- 終点: 次セグメントとのBisector
- **結果**: 始点と終点が異なる方向に移動 → 直線が歪む

### 修正: テーパー線では n = {0, 0}

```typescript
if (isTaperSegment(seg)) {
    n = { nx: 0, nz: 0 }  // 垂直オフセットなし
}
```

### ⚠️ この修正の前提条件

**成立条件**：
1. **fzが合計補正量である**
   - P側での垂直オフセット成分を含む
   - O側でfzを適用すれば完結

2. **X方向の扱い**
   - X方向の補正は別途 dx で処理
   - dx は `pToO` 関数で計算

3. **連続性の保証**
   - ノード単位でO座標が一致
   - セグメント間で不連続が発生しない

### 設計トレードオフ

**この方法の利点**：
- ✅ テーパー線の角度を保持
- ✅ 二重適用を回避
- ✅ ユーザーの手書き計算と一致

**この方法の欠点**：
- ⚠️ テーパー線とそれ以外で処理が分岐
- ⚠️ 一般性が低い（特定ケース専用）
- ⚠️ X成分の扱いが暗黙的

**代替案の検討**：

1. **ノード単位の統一dz**
   - すべてのノードでbisec.bzベースのdzを使用
   - 利点: 処理が統一される
   - 欠点: テーパー角度θ_segmentを直接使えない

2. **セグメント単位の補正**
   - セグメントごとに独立に補正計算
   - 利点: シンプル
   - 欠点: 接続点での連続性確保が困難

**採用した理由**：
実測で誤差0.000mmを達成し、角度も保持されたため。

## 原因5: 補正量の二重適用 ✅ 最も本質的

### 問題の構造

ユーザーの記事の `fz = R × (1 ± tan(θ/2))` は**合計補正量**

しかし、コードは：
1. **P座標計算**: `pz = refZ + nz × R` (垂直オフセット)
2. **O座標計算**: `oz = pz - dz` (さらにfzを適用)

**結果**: 垂直オフセット + fz が二重に適用される

### 具体例（30°テーパー）

```
垂直方向単位ベクトル: nz = -0.5

【修正前】
pz = -46 + (-0.5) × 0.8 = -46.4  (垂直オフセット)
dz = 0.586                        (fz)
oz = -46.4 - 0.586 = -46.986

合計補正: 0.4 + 0.586 = 0.986mm ❌

【修正後】
n = {0, 0}  → pz = -46           (オフセットなし)
dz = 0.586                        (fz)
oz = -46 - 0.586 = -46.586

合計補正: 0.586mm ✅
```

### 設計の分担

**P座標側の責任**：
- テーパー線: オフセットなし（n = {0, 0}）
- その他: 法線方向の基本オフセット

**O座標側の責任**：
- テーパー線: 合計補正量fzを適用
- その他: 調整補正dzを適用

**X方向の扱い**：
```typescript
// pToO関数で処理
const dx = (this.toolType === 2 || this.toolType === 1)
    ? -this.noseR
    : this.noseR

const ox = px - (dx * 2)  // X方向の補正
```

**評価**: この整理は論理的で、実測でも検証済み。

# 理論的背景（改訂版）

## 座標変換の流れ

### 1. 幾何プロファイル → P座標（中間）

```typescript
// 法線ベクトルを計算
n = getNormalAt(seg) または bisector計算

// テーパー線の場合
if (isTaperSegment) {
    n = {0, 0}  // 垂直オフセットなし
}

// P座標を計算
px = refX + n.nx * noseR
pz = refZ + n.nz * noseR
```

**P座標の意味**:
- 非テーパー: 法線方向にR だけオフセットした点
- テーパー: 幾何座標そのまま

### 2. P座標 → O座標（工具中心）

```typescript
// X方向の補正
dx = (tipNumber === 2 || tipNumber === 1)
    ? -noseR
    : noseR

// Z方向の補正
if (isTaperSegment) {
    dz = calculateDzForTaper(...)  // fz = R×(1±tan(θ/2))
} else {
    dz = calculateDzFromBisector(...)
}

// O座標を計算
ox = px - (dx * 2)  // X: 直径単位に変換
oz = pz - dz        // Z: 補正量を適用
```

**O座標の意味**: 最終的な工具中心座標

### 重要な設計決定

```
【テーパー線】
P座標: 垂直オフセットなし（幾何座標）
O座標: 合計補正量fzを適用
→ oz = 幾何Z - fz

【非テーパー線】
P座標: 法線方向にRオフセット
O座標: 調整補正dzを適用
→ oz = (幾何Z + 法線オフセット) - dz
```

この分離により、テーパー線で二重適用を回避。
