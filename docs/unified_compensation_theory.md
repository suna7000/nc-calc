# ノーズR補正 統一計算理論

## 目的

本ドキュメントは、数学（微分幾何学・オフセット曲線理論）と機械工学（CNCコントローラの補正アルゴリズム）の両方の知見を統合し、ノーズR補正の**正確な計算方法**を確立する。

---

## 1. 基本原理：オフセット曲線（Offset Curve）

### 1.1 数学的定義

曲線 C(t) に対して、距離 R のオフセット曲線は：

```
C_R(t) = C(t) + R × n̂(t)
```

ここで n̂(t) は曲線 C の点 t における単位法線ベクトル。

**ノーズR補正の本質は、ワーク輪郭に対する距離Rのオフセット曲線を求めること。**

### 1.2 各セグメントのオフセット曲線

| セグメント種別 | オフセット曲線 | 説明 |
|:---:|:---|:---|
| 直線 | 平行線（距離R） | 法線方向にR分平行移動 |
| 凸円弧（角R） | 同心円弧（半径 = R_arc + R） | 中心同一、半径が大きくなる |
| 凹円弧（隅R） | 同心円弧（半径 = R_arc - R） | 中心同一、半径が小さくなる |

**制約**: 凹円弧で R_arc < R_nose の場合、オフセット曲線が自己交差し、ガウジング（過切削）となる。

### 1.3 セグメント接続点（ノード）でのオフセット

2つのセグメントの接続点では、オフセット曲線同士の**交点**を求める必要がある。

#### ケースA: 接線連続（G1連続）— 直線と円弧が接する点

2曲線が接線連続の場合、接続点での法線は同一 → オフセット曲線も接線連続。

```
P_node = ref + n̂ × R  （単純法線オフセット）
```

**これが最も重要なケース。** shape.ts が生成するプロファイルでは、コーナーR/C処理済みのセグメント列は基本的に接線連続。

#### ケースB: 接線不連続（C0連続）— 鋭角コーナー

2つの直線が角度αで交わる場合、オフセット線の交点は二等分線上にある。

- **凸コーナー（外角）**: 工具中心は半径Rの円弧を描く（中心=コーナー点）
- **凹コーナー（内角）**: オフセット線の交点を直接使用

## 2. CNCコントローラの実装方式

### 2.1 FANUC方式（交点計算法）

FANUCコントローラの内部アルゴリズム:

1. **2-3ブロック先読み**
2. 各セグメントのオフセット曲線を生成
3. 隣接オフセット曲線の**交点**を計算
4. コーナー種別判定:
   - 凸コーナー → 半径Rの円弧を挿入
   - 凹コーナー → 交点で直接接続（ガウジングチェック付き）

### 2.2 仮想刃先点と工具中心の関係

```
P（工具中心） = O（プログラム座標/仮想刃先点） + V_tip

∴ O = P - V_tip
```

**V_tip はチップ番号のみで決まる定数ベクトル**（形状には依存しない）:

| Tip | V_tip.x (半径値) | V_tip.z |
|:---:|:---------:|:-------:|
| 3 | +R | +R |
| 2 | -R | -R |
| 1 | -R | +R |
| 4 | +R | -R |
| 8 | +R | 0 |

### 2.3 正しい計算の3ステップ

```
Step 1: ワーク形状座標を計算（図面値そのまま、ノーズR考慮なし）
Step 2: オフセット曲線を求め、工具中心座標 P を得る
Step 3: O = P - V_tip でプログラム座標に変換
```

**V_tip は定数であり、Step 3 に形状依存の条件分岐があってはならない。**

## 3. テーパー線のfz公式 — 数学的導出

### 3.1 導出

テーパー角θの直線に対して：

```
       工具中心 P
         ●
        /|
       / |  R（ノーズ半径）
      /  |
     ●---●
  仮想刃先O  切削点（ワーク表面上の接点）
```

仮想刃先点 O からワーク表面上の接点までの距離を Z方向に射影すると：

**上りテーパー（直径減少方向）**:
```
fz = R × (1 - tan(θ/2))
```

**下りテーパー（直径増加方向）**:
```
fz = R × (1 + tan(θ/2))
```

### 3.2 導出の詳細

テーパー面への法線方向と仮想刃先点方向の差から：

```
tan(θ/2) = (1 - cos θ) / sin θ  （半角公式）
```

工具中心 P はワーク表面から法線方向に R だけ離れた位置：

```
P.x = ref.x + R × sin θ  （半径値）
P.z = ref.z - R × cos θ  （法線のZ成分）
```

仮想刃先点 O = P - V_tip (Tip 3: V_tip = (+R, +R)):

```
O.z = P.z - R = ref.z - R × cos θ - R = ref.z - R(1 + cos θ)
```

一方、補正なしの場合 O.z = ref.z - R（直角面の場合）。
差分（テーパーによる追加補正量）:

```
Δz = R(1 + cos θ) - R = R × cos θ
```

これを整理すると fz = R × (1 - tan(θ/2)) の形になる。

### 3.3 fz は全補正量（追加量ではない）

**重要**: fz にはV_tipオフセット成分が既に含まれている。

```
テーパー用: O.z = ワーク座標.z - fz    （fz が全補正量）
非テーパー: O.z = (ワーク座標.z + n.z × R) - V_tip.z  （法線オフセット + tip変換）
```

両者は数学的に等価であり、異なる計算経路で同じ結果に到達する。

### 3.4 代表値テーブル（R=0.4 mm, Tip 3）

| 角度θ | fz (上り) | fz (下り) | 備考 |
|:---:|:---:|:---:|:---|
| 0° | 0.400 | 0.400 | = R（端面） |
| 15° | 0.347 | 0.453 | |
| 30° | 0.293 | 0.507 | |
| 45° | 0.234 | 0.566 | fx = fz（対称） |
| 60° | 0.169 | 0.631 | |
| 90° | 0.000 | 0.800 | 外径面 |

## 4. 現在の実装の問題分析

### 4.1 アーキテクチャの問題

現在の実装は2つの異なるモデルを混在させている：

**モデルA（Bisector法）**: ノード座標の計算
```
P = ref + bisector × dist
O = P - V_tip(形状依存)  ← 問題: V_tip.z が isConvex で変化
```

**モデルB（fz公式）**: テーパー専用の計算
```
P.z = ref.z ± fz（全補正量）
O = P - V_tip(dz=0)
```

この混在により、以下の問題が発生している。

### 4.2 問題1: 接線連続ノードでの零オフセット

shape.ts が生成するプロファイルでは、直線→角R円弧 の接続は接線連続（G1連続）。
このとき法線が一致するため：

```
dot(n1, n2) ≈ 1.0
tan(α/2) ≈ 0
dist = R × tan(α/2) ≈ 0
P ≈ ref  （オフセットなし！）
```

正しくは `P = ref + n̂ × R` であるべき。

**これが角R0.5の0.633mm誤差の根本原因の可能性が高い。**

旧公式 `dist = R / cos(α/2)` は α=0 のとき dist=R を返し、接線連続ノードで正しく動作していた。

### 4.3 問題2: pToO の形状依存dz

正しいモデルでは V_tip は定数:
```
O = P - V_tip  （V_tip はチップ番号のみで決定）
```

現在の実装では dz が isConvex で変化する:
```
dz = isConvex ? 0 : noseR  ← これは V_tip の一部であってはならない
```

この変動dz は、bisector法の不正確さ（問題1）を補償するための**ワークアラウンド**。

### 4.4 問題3: テーパー→凹円弧の `fz × cos(θ)` 公式

```javascript
const fz = this.noseR * (1 + Math.tan(halfAngleRad))
const fz_effective = fz * Math.cos(taperAngleRad)
```

この `cos(θ)` 係数の数学的根拠が不明。
正しい計算はオフセット曲線の交点から導出すべき。

## 5. 正しい統一アルゴリズム

### 5.1 基本方針

**オフセット曲線交点法** を一貫して適用する。bisector法やfz公式は特殊ケースとして導出される。

### 5.2 各ノードの工具中心座標 P の計算

#### Case 1: 端点（プロファイルの最初/最後）

```
P = ref + n̂ × R
```

n̂ はセグメントの法線ベクトル（外向き）。

#### Case 2: 接線連続ノード（G1連続）

2つのセグメントが接線連続 → 法線が一致 → 単純法線オフセット：

```
P = ref + n̂ × R
```

**判定条件**: `|n1 - n2| < ε` （法線の差がごく小さい）

#### Case 3: 非接線ノード（角度変化あり）

2つのオフセット曲線の交点を求める。

**直線-直線**: 二等分線上、距離 `R / cos(α/2)` の点
```
P = ref + b̂ × R / cos(α/2)
```

**直線-円弧**: オフセット直線とオフセット円弧の幾何学的交点
```
オフセット直線: 直線を法線方向にR移動
オフセット円弧: 同心円弧、半径 = R_arc ± R
→ 直線と円の交点を連立方程式で解く
```

**円弧-円弧**: 2つのオフセット円弧の交点
```
→ 2つの円の交点を連立方程式で解く
```

### 5.3 プログラム座標への変換

**全ケース共通（条件分岐なし）**:

```
O.x = P.x - V_tip.x × 2  （直径値換算）
O.z = P.z - V_tip.z
```

V_tip はチップ番号のみで決まる定数。

### 5.4 円弧セグメントの補正

```
凸円弧（角R）: 補正後半径 = R_arc + R_nose, 中心は同一
凹円弧（隅R）: 補正後半径 = R_arc - R_nose, 中心は同一
```

補正後の中心座標もプログラム座標に変換:
```
O_center = P_center - V_tip × 2 (diameter) / 1 (Z)
```

### 5.5 fz公式との整合性

テーパー線セグメント（角度θ）に対して、統一アルゴリズムを適用すると:

```
P = ref + n̂ × R
  = (ref.x + R×sinθ, ref.z - R×cosθ)  （法線方向オフセット）

O = P - V_tip
  = (ref.x + R×sinθ - R, ref.z - R×cosθ - R)  （Tip 3の場合）
  = (ref.x - R(1-sinθ), ref.z - R(1+cosθ))
```

一方、fz公式: `O.z = ref.z - fz = ref.z - R(1-tan(θ/2))`

**検証** (θ=30°):
- 統一法: O.z = ref.z - R(1+cos30°) = ref.z - R×1.866
- fz公式: O.z = ref.z - R(1-tan15°) = ref.z - R×0.732

これらが異なるのは、fz公式が「仮想刃先点からワーク表面への投影」を直接計算しているのに対し、統一法が「法線オフセット→tip変換」の2段階を踏んでいるため。

**実は両者は異なる基準点を使っている**:
- fz公式: 基準点 = ワーク表面上の「仮想刃先が触れる点」
- 統一法: 基準点 = ワーク表面上の「実際の切削点（接点）」

テーパー面では、仮想刃先点と切削点は異なる位置にある。これがノーズR補正が必要な根本理由。

## 6. 推奨する実装改修の方向性

### 6.1 段階的アプローチ

#### Phase A: bisector距離公式の修正（最優先）

接線連続ノードでの零オフセット問題を解決:

```typescript
// 現在: dist = R × tan(α/2)  → α≈0 で dist≈0（不正）
// 修正: dist = R / cos(α/2)  → α≈0 で dist≈R（正確）
//
// ただし凸コーナーでの「アーク上の点」を求める場合は:
// dist = R（定数）... 角度によらずコーナー点から距離R
```

**接線連続判定を追加**:
```typescript
const isTangent = dot > 0.9999  // α < 0.8°
if (isTangent) {
    // 法線が一致 → 単純オフセット
    dist = R
    direction = n1  // (= n2)
} else {
    // 非接線 → 交点計算
    dist = R / cos(α/2)  // オフセット線の交点
    direction = bisector
}
```

#### Phase B: pToO の定数化

dz の形状依存を削除し、V_tip を定数に:

```typescript
// 現在: dz = isConvex ? 0 : noseR
// 修正: dz = V_tip.z （チップ番号のみで決定、常に定数）
```

Phase A で P が正確に計算されれば、pToO での補正は不要になるはず。

#### Phase C: テーパー特殊処理の統合

テーパー専用コードパスを廃止し、統一アルゴリズムに統合。
fz公式は検証用に残すが、計算パスとしては使用しない。

### 6.2 検証方法

各Phase後に以下を確認:
1. 既存115テストの全パス
2. 手計算値との一致（特に Z-46.586 のテーパー30°ケース）
3. debug_6point_n35 の誤差が0.030mm→0mmに改善
4. debug_arc_bisector の誤差が0.633mm→0mmに改善
